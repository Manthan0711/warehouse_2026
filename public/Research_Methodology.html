3. Phase 2 — Data Preprocessing & Cleaning

The raw WDRA CSV data required extensive preprocessing to transform government registry records into ML-ready feature vectors. This phase involved four sequential sub-pipelines implemented across multiple scripts.

3.1 Preprocessing Pipeline Architecture

1
CSV Parsing & Initial Validation (scripts/csv-to-json.js)
Raw CSV rows are parsed using the csv-parse/sync library. Each of the 12 columns is mapped to standardized internal field names. Records with missing critical fields (WH Name, Capacity, District) are flagged.
2
Pincode Extraction & Validation
Pincodes are extracted via regex pattern /(\d{6})/ from the address field. Invalid or missing pincodes are imputed using a city-specific pincode lookup table containing 500+ verified Maharashtra pincodes across 30+ cities (scripts/fix-warehouse-data-quality.js).
3
Geospatial Coordinate Imputation
The raw WDRA data contains no latitude/longitude coordinates. Coordinates are imputed using a centroid-based approach: each of the 30+ Maharashtra cities/districts has a known centroid (e.g., Mumbai: 19.0760°N, 72.8777°E; Pune: 18.5204°N, 73.8567°E). A random Gaussian offset of ±0.05° (~5–10 km) is applied to simulate realistic geographic distribution:
latimputed = latcentroid + 𝒩(0, 0.05)      lngimputed = lngcentroid + 𝒩(0, 0.05)
4
Contact Data Cleaning
Placeholder emails (@example.com) are replaced with realistic generated emails. Invalid phone numbers (scientific notation artifacts like "9.67E+09") are regenerated using valid Indian mobile prefixes (+91 7x/8x/9x followed by 8 digits).
5
License Expiry Validation (shared/ml-algorithms.ts)
Each warehouse record is validated for license expiry using the checkLicenseExpiry() function. Warehouses with expired licenses are excluded from recommendation pools. Those expiring within 3 months receive a degraded health score: score = 0.3 + 0.7 × (daysRemaining / 90).

4. Phase 3 — Feature Engineering & Synthetic Data Augmentation

4.1 Feature Augmentation from 12 to 33 Attributes

The raw WDRA dataset contains only 12 attributes — insufficient for training a multi-dimensional recommendation engine. The following 21 features were synthetically engineered based on domain knowledge, WDRA regulatory norms, and Maharashtra market data:

# Engineered Feature Type Derivation Logic
1 pricing (₹/sqft/month) Numeric Market research: ₹25–150 range, type-dependent
2 total_size_sqft Integer Capacity (MT) × 10 conversion factor + noise
3 occupancy Float [0–1] Random sample [0.30, 1.00] with type-weighted mean
4 rating Float [3.0–5.0] Composite: 3.5 base + certificate(0.8) + status(0.5) + occupancy(0.4) + type(0.3)
5 reviews_count Integer (capacity / 1000) × rating × random(1, 3)
6 amenities[] String Array Type-dependent: Cold Chain → ["Temperature Control", "Refrigeration", "Climate Monitoring"]
7 warehouse_type Categorical 8 types: General, Cold Chain, Pharma, FMCG, Industrial, Zepto Dark Store, etc.
8 ownership_certificate Binary Random: 80% Verified, 20% Pending
9 image_url URL String Type-mapped Unsplash warehouse images
10 description Text Template-based: "{type} facility in {district} with {capacity} MT capacity..."
11 micro_rental_spaces Integer Parsed from extended CSV or estimated from total_area
12 latitude / longitude Float Centroid-based imputation with Gaussian offset
13 total_blocks Integer ⌈total_size / 1000⌉ — spatial grid blocks
14 available_blocks Integer ⌈total_blocks × (1 - occupancy)⌉
15 grid_rows / grid_cols Integer ⌈√total_blocks⌉ for visual grid layout
16–21 power_backup, cold_storage, fire_safety, cctv, loading_docks, parking Boolean / Integer Type-dependent assignment: loading_docks = ⌊area / 10000⌋

4.2 Synthetic Data Augmentation: 424 → 10,002 Records

To achieve a statistically meaningful dataset for ML training and realistic platform demo, the 424 WDRA-sourced records (5 seed records with full detail) were augmented to 10,002 records through programmatic generation (client/data/enhanced-warehouses.ts):

Augmentation Strategy

  • 5 detailed seed records: Hand-crafted from real WDRA data with full attribute set
  • 149,995 generated records: Programmatic generation via generateWarehouseData() loop
  • Distribution preservation: District, type, and capacity distributions mirror WDRA patterns
  • 10 districts: Mumbai, Pune, Nashik, Aurangabad, Solapur, Kolhapur, Sangli, Satara, Thane, Raigad
  • 7 warehouse types: Zepto Dark Store, Pharma Cold Chain, Industrial Logistics, Automobile, Textile, Food, General
  • Database population: 10,002 records loaded into Supabase PostgreSQL via batch SQL INSERT with ON CONFLICT upsert

4.3 SQL Data Loading Pipeline

6. Phase 5 — Machine Learning Algorithms (12-Algorithm Ensemble)

The SmartSpace recommendation engine implements 12 distinct ML algorithms across two implementation files (shared/ml-algorithms.ts — 1,563 lines, and shared/advanced-ml-algorithms.ts — 579 lines), organized into two ensemble systems:

6.1 Primary Ensemble: 5-Algorithm Advanced System

File: shared/ml-algorithms.ts — Used as the default recommendation algorithm (?algorithm=ensemble).

Algorithm 1: K-Nearest Neighbors (KNN)

Hyperparameters: K=8, 8-dimensional weighted Euclidean distance

Feature Vector (8 dimensions):

F = [locationScore, priceScore, areaScore, typeScore, 
     verificationScore, availabilityScore, ratingScore, amenitiesScore]

Feature Weights (preference-adaptive):

Feature Weight (district specified) Weight (no district)
locationScore 6.0 (dominant) 0.5
priceScore 2.5 0.5
areaScore 2.0 0.5
typeScore 1.5 0.5
ratingScore 1.0 1.0
amenitiesScore 0.8 0.8

Distance Metric:

d(x, xideal) = √( Σi=1..8 wi · (xi - xideal,i)² ) / √(Σ wi)

similarity = 1 − dnormalized

Algorithm 2: Random Forest

Hyperparameters:

  • Number of trees: 50 (n_estimators = 50)
  • Bootstrap sampling rate: 80% per tree
  • Feature bagging: √n features per tree (≈3 of 9 features)
  • 9 feature extractors: locationMatch, priceMatch, areaMatch, typeMatch, verificationMatch, availabilityMatch, rating, amenitiesQuality, licenseHealth

Aggregation: Weighted majority voting across 50 decision trees

Feature Importance (adaptive):

importancelocation = 0.50 (if district specified) | 0.05 (otherwise)
importanceprice = 0.25 | importancearea = 0.20 | importancetype = 0.15
importancerating = 0.10 | importanceamenities = 0.08

Algorithm 3: Gradient Boosting

Hyperparameters:

  • Iterations: 10
  • Learning rate (η): 0.1
  • Weak learners: Single-feature decision stumps

Update Rule:

Fm(x) = Fm-1(x) + η · hm(x)
where hm = argminh Σ L(yi, Fm-1(xi) + h(xi))

Algorithm 4: Neural Network (Multi-Layer Perceptron)

Architecture: 10 → 10 → 4 → 1 (fully connected)

  • Input layer: 10 features (all 9 feature scores + composite)
  • Hidden layer 1: 10 neurons, Leaky ReLU activation
  • Hidden layer 2: 4 neurons, Leaky ReLU activation
  • Output layer: 1 neuron, Sigmoid activation → [0, 1] score

Activation: Leaky ReLU: f(x) = max(0.01x, x)

Algorithm 5: Hybrid Stacking

A meta-learner that combines KNN and Random Forest outputs as input features to a final scoring layer.

Ensemble combination: Weighted average of base learner predictions with learned meta-weights.

5-Algorithm Ensemble Aggregation

Scoreensemble = Σa ∈ {KNN, RF, GB, NN, Hybrid} wa · Scorea
where dynamic weight wa ∝ confidencea

6.2 Secondary Ensemble: 5-Algorithm Advanced ML System

File: shared/advanced-ml-algorithms.ts — Used when ?algorithm=hybrid is specified.

Algorithm Key Parameters Ensemble Weight
KNN (Gaussian Kernel) K=15, σ=0.8, 6-dim feature vector 0.25
Content-Based Filtering (TF-IDF-like) Weights: location=0.35, price=0.25, area=0.20, type=0.10, quality=0.10 0.25
Collaborative Filtering Simulated user behavior patterns, popularity-demand-price model 0.20
Neural Network (Sigmoid MLP) 6→3→2→1 architecture, sigmoid activation at all layers 0.15
Bayesian Scoring Prior × Likelihood / Evidence, simplified posterior estimation 0.15

Gaussian Kernel Similarity (KNN Advanced)

K(d) = exp(−d² / 2σ²)    where σ = 0.8

Bayesian Posterior Estimation

P(match | evidence) = P(evidence | match) · P(match) / [P(evidence | match) · P(match) + P(evidence | ¬match) · P(¬match) + ε]
where ε = 0.01 (Laplace smoothing)

Confidence Calculation

confidence = 1 − min(σscores, 0.3) / 0.3
where σscores = std. deviation of 5 algorithm scores

7.2 LLM Call Flow & Fallback Chain

The system implements a priority-ordered fallback chain across two gateway files. Groq (Llama 3.3 70B) is always attempted first — the other providers exist purely as fault-tolerance backups and are never reached when the Groq API key is configured (which it always is in production).

Client-Side Gateway: client/services/aiService.ts

Consumers: GeminiChatbot.tsx, PricingRecommendationModal.tsx, ListProperty.tsx, OwnerNotificationsPage.tsx, Dashboard.tsx, smartBookingService.ts, warehouseService.ts 

const providers = [
  { name: 'Groq (Llama 3.3)',  fn: callGroq,       enabled: !!GROQ_API_KEY },        // ← ALWAYS FIRST
  { name: 'OpenRouter (Claude)', fn: callOpenRouter, enabled: !!OPENROUTER_API_KEY },  // Backup only
  { name: 'Cloudflare AI',      fn: callCloudflare, enabled: !!(CF_ACCOUNT && CF_TOKEN) },
  { name: 'Gemini Pro',         fn: callGemini,     enabled: !!GEMINI_API_KEY }        // Last resort
];
// Iterates in order; returns on first success

Server-Side Gateway: shared/advanced-llm-service.ts

Consumer: server/routes/recommend.ts (LLM recommendation endpoint) 

Fallback chain: callGroq()  →  callOpenRouter()  →  callGemini()
Model strings:  llama-3.3-70b-versatile  →  claude-3-haiku  →  gemini-1.5-flash

7.5 Prompt Engineering — Warehouse Recommendation

The LLM recommendation prompt follows a structured template with mathematical scoring criteria: 

System: "You are an expert warehouse recommendation AI. Always respond with valid JSON only."

User Prompt Structure:
1. USER PREFERENCES (JSON): district, targetPrice, minArea, preferredType
2. WAREHOUSE DATA: Top 50 candidates (id, name, district, price, area, rating, type)
3. SCORING CRITERIA:
   - Location Match (30% weight): exact=1.0, region=0.7, different=0.3
   - Price Match (25% weight): within budget=1.0, +10%=0.8, +20%=0.6
   - Size Match (25% weight): meets minimum=1.0, within 10%=0.8
   - Quality (20% weight): rating≥4.5=1.0, ≥4.0=0.8, verified=+0.1
4. OUTPUT FORMAT: Structured JSON with warehouseId, score, reasoning, matchFactors

Key Novelty

This research presents a hybrid ML + LLM recommendation architecture where:

  • ML algorithms (12 total) handle the computational heavy-lifting of scoring 10,000+ warehouses using mathematical distance metrics, ensemble voting, and neural network inference — providing speed and scalability.
  • Meta Llama 3.3 70B provides natural language reasoning, contextual understanding, and human-readable explanations for recommendations — bridging the gap between numerical scores and user-interpretable insights.
  • The two systems operate in a complementary fashion: ML processes the full candidate pool (10,000 records), while the LLM re-ranks a curated subset (top 50) with deeper semantic analysis.
